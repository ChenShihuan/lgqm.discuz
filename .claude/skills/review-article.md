# Review Article — 审阅优化同人文章

## 描述

读取 import 生成的原始 .raw.mw 文件，逐步审阅优化，输出最终版 .mw 文件。可与原始文件对比差异。

## 触发条件
- 要求审阅某文章时，调用此技能
- "审阅 <文章名>"
- "review <文章名>"
- "review-article <文章名>"
- import-article 完成后

## 执行流程

### Step 0: 确定目标文件

- 如果用户指定了文件路径，直接使用该文件
- 如果用户在 IDE 中选中了某个 `.raw.mw` 文件，以选中文件为目标
- 否则，扫描 `output/` 目录下的 `.raw.mw` 文件，列出供用户选择

```bash
find output -name "*.raw.mw" 2>/dev/null
```

设目标文件为 `TARGET`（如 `output/22085-淞沪启明同人/text/淞沪启明同人.raw.mw`），`NAME` 为文件名，`TID` 可从路径中提取。

### Step 1: 查看审阅项

```bash
python3 -m monitor.cli review-info <TARGET>
```

输出：空白 Infobox 字段、章节标记（是否已格式化）、同人注释数、残留检查、重复检测。

### Step 2: 逐项优化 .mw 文件

对 `TARGET` 执行以下操作（如目标是 `.raw.mw`，则输出文件为对应的 `.mw`）：

**2a. Infobox 补全（必须）**
- 提取故事发生地 → 填写 `| 地点 =`
- 分析内容涉及领域 → 填写 `| 涉及方面 =`（工业、军事、航运、农业等，中文顿号分隔）
- 提取关键词 → 填写 `| 内容关键字 =`
- 如有下载图片 → 填写 `| 图像 = [[Image:xxx.jpg|class=img-responsive]]`
- 确认 `| 官方论坛 =` 不含 `<!--作者ID-->` 注释

**2b. 章节标题格式化（必须）**
- 查找「第X章」「序章」「终章」「尾声」等标记
- 格式化为 `=== 第X章 标题 ===`（3 级标题）
- 章节超过 3 节时，在 `{{首行缩进start}}` 后添加 `__TOC__`
- **论坛 raw 格式标题**：`'''N.标题'''` 或 `'''终章.xxx'''` → 转换为 `== 标题 ==`（2 级标题），下篇新增章节用 `== ==` 与上篇/下篇的 `= =` 区分层级
- 上篇/下篇用 `= =`（1 级），章节用 `== ==`（2 级），小节用 `=== ===`（3 级）

**2b-ii. 合并被误拆的连续章节**

导入时 Python 脚本已自动消除对话开头、超长标题等明显误判（预期减少 ~40% 虚假章节）。审阅时仍需检查以下情况：

- **短章节合并**：连续两个 `== ==` 标题之间正文不足 500 字 → 很可能应合并
- **长标题检查**：章节标题超过 30 字 → 人工判断是否是真正标题还是内容首行被误用
- **无空行相邻**：两个章节之间无 `{{同人注释start}}` 或明显场景切换 → 检查是否应合并
- **角色延续**：两个连续章节标题包含同一角色名 → 可能是同一场景的连续发帖

合并操作：删除第二个 `== 标题 ==`，在正文之间插入空行分隔。

**2c. 同人注释边界检查**
对于同人注释，应当根据**同人注释规则**进行检查
- 确认作者正文未被误包裹在 `{{同人注释start}}...{{同人注释end}}` 中
- 检查注释块内是否有作者回复 → 提取回正文
- 清理注释块内的「本帖最后由」「发表于」残留
- 检查 `{{同人注释end}}` 后是否有重复内容 → 删除

**2d. 段落格式化与清理**

论坛导入的文章常见以下格式问题，需逐项修复：

1. **首行缩进空格**：论坛用 4 个 `&nbsp;`（被 converter 转为 4 个普通空格）作为段落缩进。在 MediaWiki 中由 `{{首行缩进start}}` 模板处理，这些空格应转为段落分隔 `\n\n`：
   ```python
   # 正文区域内 2+连续空格 → 段落分隔
   body = re.sub(r' {2,}', '\n\n', body)
   ```

2. **邻行合并**：论坛每行一个段落，但导入后相邻非空行之间缺少空行。需在每对相邻非空行之间插入空行：
   ```python
   for i, line in enumerate(lines):
       result.append(line.rstrip())
       if i < len(lines)-1 and line.rstrip() and lines[i+1].rstrip():
           result.append('')
   ```

3. **常规清理**：
   - `\xa0` (non-breaking space) → 删除（保护 Infobox 区域）
   - `[[Image:static/image/smiley/...]]` → 删除（论坛内置表情，非文章配图）
   - `[[Image:static/image/common/...]]` → 删除（论坛 UI 图标）
   - 连续空行 > 3 → 压缩为 2
   - **长等号分隔线**：`=======`（≥5 个 `=`）会被 MediaWiki 解析为 h5/h6 标题，造成目录污染。应转为短横线 `---------`（9 个 `-`）：
     ```python
     # ≥5 个连续 = → 转为 9 个 -
     content = re.sub(r'^={5,}\s*$', r'---------', content, flags=re.MULTILINE)
     ```
   - `.raw.mw` 残留的 `{{PAGENAME}}` 占位符 → 替换为文章名
   - `<!--作者ID-->` 注释 → 删除
   - **纯编码图片文件名残留**：`[[File:xxx.jpg]]` 后紧跟的 `'''文件名'''` 仅为无意义的原始文件名文本残留，应删除，仅保留 `[[File:...]]`。涵盖两类：
     1. **Hash 文件名**：`6c224f4a20a44623bcb020840b8931090df3d715.jpeg`（32-40 位 hex）
     2. **截图/相机标准命名**：`暴风截图202192410247281.jpg`、`微信图片_20230322001727.jpg`、`IMG_20230101.jpg`、`DSC0001.jpg`、`Screenshot_2023.png` 等
     ```python
     # 删除 [[File:xxx.jpg]] 后紧跟的 '''无意义文件名''' 残留
     import re
     patterns = [
         r'[0-9a-f]{32,40}',              # hash 文件名
         r'(暴风截图|微信图片|IMG_|DSC|Screenshot_|Photo_|PANO_)\S*',  # 截图/相机命名
     ]
     for p in patterns:
         content = re.sub(
             rf'\[\[File:([^\]]+)\]\]\s*\n\s*\'\'\'{p}\.[a-z]+\'\'\'',
             r'[[File:\1]]',
             content
         )
     )
     ```

4. **同人注释质量过滤**：见下方「同人注释规则」章节。

**2e. 目录/楼层索引清理**

首楼（或某独立楼层）的纯目录列表应删除，避免与正文混淆。常见格式：

- **`XX楼 关于XXX`**：楼层号+标题的无链接列表（如主攻方向）
- **`XX楼更新第X章`**：楼层→章节名映射（如面首首楼 TOC）
- **`[url pid=数字 章节名]`**：含 PID 链接的目录（如太平无战事、澳宋大调查二楼）
- **`第X回 / 第一回 ...`**：编号章节列表（如潮州陷落、崇祯·1644）

清理方法：
```python
# 删除首楼 TOC 块（从"目录"或第一个章节列表行开始，到正文内容之前）
# 如果 TOC 是独立楼层（如二楼），在 toc_analysis.json 中设 source_floor 自动跳过
# 如果 TOC 嵌入首楼且无"目录"关键字，识别规则：连续 5+ 行章节格式文本
import re
toc_pattern = re.compile(r'^(第[一二三四五六七八九十\d]+[回章节].+|[\d]+楼\s+.+|\[https?://[^\]]+pid=\d+[^\]]+\])$', re.MULTILINE)
```

**2f. 章节标题批量修复**

针对 converter 未识别但实为章节标题的文本模式，使用批量替换：

| 原始格式 | 正则 | 替换为 |
|---------|------|--------|
| `XX，标题`（中文数字） | `^([一二三四五六七八九十百千]+)[，,、]\s*(.+)$` | `== \1，\2 ==` |
| `'''第X章 标题'''`（粗体） | `^'''第.+章\s+.+'''$` | `== 标题 ==`（手动提取） |
| `第X章 标题`（纯文本） | `^(第[一二三四五六七八九十\d]+[回章节].+)$` | `== \1 ==` |
| `间章X 标题` / `番外 标题` | `^(间章[一二三四五]?\s+.+\|番外[·•\s].+)$` | `== \1 ==` |
| `《临高启明》同人《面首·X》` | `^《临高启明》同人《(.+)》$` | `== \1 ==` |
| 嵌套子章节（第X日/上中下） | `^== (第十[四五六]章：[^=]+) ==$` | `=== 子节标题 ===` |

批量修复脚本：
```python
import re
patterns = [
    # 中文数字章节: "五十一，发票" → "== 五十一，发票 =="
    (r'^([一二三四五六七八九十百千]+)[，,、]\s*(.{1,30})$', r'== \1，\2 =='),
    # 标准回目: "第一回 标题" → "== 第一回 标题 =="
    (r'^(第[一二三四五六七八九十\d]+回\s+.+)$', r'== \1 =='),
    # 间章: "间章一 标题" → "== 间章一 标题 =="
    (r'^(间章[一二三四五]?\s+.+)$', r'== \1 =='),
]
for pat, repl in patterns:
    content = re.sub(pat, repl, content, flags=re.MULTILINE)
```

**2g. 嵌套章节层级规则**

当文章有章节嵌套（卷/案→章→节）时，应用层级：

| level | 标记 | 场景 |
|-------|------|------|
| 1 | `= xxx =` | 卷/案/篇（顶层分组，如"一 南海案"、"第一卷"） |
| 2 | `== xxx ==` | 标准章节（默认，大多数章节） |
| 3 | `=== xxx ===` | 子章节/小节/按日分节 |

判断嵌套的方法：
- 编号体系：顶层"一 南海案"后跟子编号"一""二""三" → 顶层 level=1，子编号 level=2
- "第X章"含多日分节（第八日/第九日）→ 章 level=2，日 level=3
- "第X章"有上中下 → 章 level=2，上/中/下 level=3

**2h. 非正文内容清理**

审阅时需检查并清理以下非正文内容：

1. **作者碎碎念**："最近孩子身体不舒服，没写...""ps，想和各位聊聊..." → 删除或移入同人注释
2. **收款码/打赏链接**：`.jpg` 结尾的收款码图片 → 删除
3. **论坛签名/无关宣传**：如 AI 有声书链接、个人广告 → 删除
4. **附录/法规引用**：如铁路道口管理办法（20条法规）→ 标注为 `== 附录：标题 ==`
5. **"写给各位元老"类说明** → `{{同人注释start}}...{{同人注释end}}`

**2i. 文章名修正**

导入后检查 Infobox `| 同人作品 =` 是否含残留后缀。clean_article_name 已知无法处理的格式：

- `【磺胺及青蒿素已点亮】` 等自定义标签在标题末尾 → 手动删除
- 含 `【原创+搬运+完结】` 多重前缀 → 前缀循环剥离已处理
- `YYYY.M.D更新` 含4位年份的括号日期 → 已修复（2024.1.9更新）

对已有 Wiki 文章执行 update 后，需额外检查：

1. **定位 Wiki 最后一章在论坛原文中的位置**：取 Wiki 最后 200 字的指纹，在 fresh import 中搜索匹配 → 匹配位置之后的内容才是真正新增
2. **删除重复章节**：fresh import 中 Wiki 已有章节（标题不同但内容相同）需识别并跳过
3. **仅插入新增内容**：将 fresh import 中 Wiki 末章之后的新章节追加到 Wiki 原文末尾
4. **新章节标题格式化**：按 2b 规则转换 raw 标题

**2j. 文章完结情况检查**

导入后根据总体予以判断文章的完结情况。如有以下情况，应当被视为完结：

- 文章包含`全文完`、`（完）`等结尾。
- 文章包含`尾声`、`后记`等章节。
- 对于中短篇文章，全文在故事性上完整，以开放性、展望性场景描述结束。

判断完结后，应当修改 Infobox `| 完结情况 = ` ，更新为`完结`。


### Step 3: 对比差异

```bash
diff -u <TARGET> <TARGET_DIR>/<NAME>.mw | head -200
```

同时输出变更摘要：章节数变化、同人注释数变化、段落数变化、文件大小变化。

## 同人注释规则

### 引用回复讨论转换到同人注释

引用回复讨论的内容，不需要全部保留为注释，需要内容判断其重要性，仅具有较重要的补充说明价值的内容才需要补入注释，补入注释时可以进行精简、总结。

**以下无实质内容的回复应直接删除，不保留为注释：**
- 纯催更/赞美类：如「赞美更新」「催更」「加油」「顶」「支持」「楼主加油」「前排」「马克」「留名」「先赞后看」等
- 纯表情回复：`[[Image:static/image/smiley/...]]` 仅为 Discuz 默认表情，不是文章配图 → 删除
- 纯标点/纯数字/纯英文短句（如「666」「hhh」「...」）→ 删除
- 长度 < 10 字且无实质信息的回复（如「写得好」「不错」「等更」）→ 删除
- 仅重复正文内容、无补充信息的回复 → 删除
- 同一用户重复发相同/高度相似内容 → 去重，保留一条

**同人注释批量过滤脚本：**
```python
def should_keep(text):
    clean = re.sub(r'\[\[Image:[^]]+\]\]', '', text).strip()
    clean = re.sub(r'\n+', ' ', clean).strip()
    if not clean: return False
    if len(clean) < 10: return False
    praise = ['赞美更新','催更','加油','顶','支持','楼主加油','前排','马克','留名','先赞后看','写得好','等更','好康']
    for pw in praise:
        if pw in clean and len(clean) < 25: return False
    if re.match(r'^[\s!！。.…\-—~～0-9a-zA-Z]+$', clean) and len(clean) < 15: return False
    return True
```

**评论需有补充价值才保留：**
- 提供背景知识/考据/纠错 → 保留并精简
- 剧情讨论/人物分析/发展建议 → 保留并精简
- 作者对读者问题的回复 → 保留在正文（不放入注释）
- 图片说明/配图 → 按图片规则处理

```
作者正文（首楼 + 作者后续帖）  →  保留为正文
其他用户回复/讨论（有补充价值）→  {{同人注释start}}...{{同人注释end}}
其他用户回复（无实质内容）     →  直接删除
「XXX 发表于 HH:MM」           →  自动删除
作者引用回复他人的内容          →  保留在正文
```

示例：
```
=== 第五章 清剿收编 ===
第二天一早，天刚亮，杨兵和赵刚就带着人出发了。

作者紧接着继续的正文内容...

（讨论注释放在此章节最后）

{{同人注释start}}
当年还没横沙岛啊，不如转到洋山港去。
{{同人注释end}}


=== 第六章 XXXX === （作者紧接着继续的正文内容下一章节）
第二天一早，天刚亮，杨兵和赵刚就带着人出发了。

作者紧接着继续的正文内容...
```
### 图片到同人注释

对于文章插图，应当根据图片说明判断，有必要转换到同人注释的，应当插入在原始的插入处，如下：

示例：
```
{{同人注释start}}
当初穿越者带来的四艘[[8154型单拖网渔船| 8154型渔轮]]，在临高海军的序列里是以“8154型高速巡洋舰”的名称入籍的。随着时间的推移，这四条渔轮的老化、失灵问题越来越严重。

[[File:8154结构图.jpg|600px]]
{{同人注释end}}
```

如果仅有简要的说明的配图，可以参考以下示例，直接插入在原文，不需要使用同人注释标签

示例：
```
[[image:095617tkotqng4tbzxjjxg.jpg|400px|class=img-responsive|thumb|来几张盖伦和卡拉克同框]]

[[image:095617w66br46z626b4mdq.jpg|400px|class=img-responsive|thumb|来几张盖伦和卡拉克同框]]

[[image:095617fnhdqhg606zdbg6x.jpg|400px|class=img-responsive|thumb|来几张盖伦和卡拉克同框]]
```
